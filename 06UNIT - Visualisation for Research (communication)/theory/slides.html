PART I

Visualisation Principles

What makes a good graphic?

Part I covers the theoretical foundations ‚Äî Tufte's principles, perception science and the Grammar of Graphics.

Tufte's Principle #1: Data-Ink Ratio

Maximise Data-Ink

Every drop of ink (every pixel) should convey information. Remove anything that doesn't serve the data.

Data-Ink Ratio = Data-Ink / Total Ink Used

‚ùå Poor

  • Heavy gridlines
  • Unnecessary borders
  • Gratuitous 3D effects
  • Redundant legends

‚úì Good

  • Subtle or no gridlines
  • Direct labels on data
  • 2D when sufficient
  • Deliberate minimalism
The data-ink ratio is Tufte's most famous principle. Every visual element should justify its existence.

Tufte's Principle #2: Lie Factor

Avoid Visual Deception

Lie Factor = (Visual Effect) / (Data Effect)

Ideal: Lie Factor = 1.0
Acceptable: 0.95 ‚Äì 1.05
Problematic: < 0.67 or > 1.5

Classic manipulation: Y-axis not starting at 0. A 2% increase can appear as 200% if the axis runs from 98 to 102.

The lie factor quantifies visual dishonesty. Show examples of truncated axes in popular media.

Tufte's Principle #3: Small Multiples

Repeat the Design, Vary the Data

When comparing across categories or time, use small multiples: identical chart designs with different data slices.

Tufte's Principle #4: Micro/Macro Readings

Support Multiple Levels of Detail

A great visualisation works at multiple scales:

  • Macro: Overall pattern visible at a glance
  • Micro: Individual data points accessible on inspection

Example: A scatter plot showing overall correlation (macro) while allowing identification of specific outliers (micro).

Interactive visualisations excel at micro/macro readings through zoom and hover interactions.

Visual Perception: Cleveland-McGill Hierarchy

Not all visual encodings are equally effective. Accuracy ranking:

RankEncodingBest For
1Position on common scalePrecise comparison
2Position on aligned scalesGrouped comparison
3LengthBar charts
4Angle / SlopeTrends (with caution)
5AreaRough magnitude
6Volume / Colour saturationCategorical grouping

‚ö†Ô∏è Pie charts use angle ‚Äî ranked 4th. Consider bar charts instead!

This hierarchy is based on perceptual psychology research. It explains why bar charts outperform pie charts for precise comparisons.

Colour: Accessibility First

~8% of males have colour vision deficiency. Design for everyone.

Colourblind-Friendly Palette (Wong, 2011)

#0072B2 #E69F00 #009E73 #CC79A7 #F0E442 #56B4E9 #D55E00

Sequential data:
Use perceptually uniform: viridis, cividis
Diverging data:
Use coolwarm, RdBu (red-blue)
The Wong palette is specifically designed for colour vision deficiency. Viridis is perceptually uniform and prints well in greyscale.

Question

A bar chart's Y-axis starts at 50 instead of 0. The bars show values of 52, 54 and 58. What is the approximate Lie Factor for the difference between the smallest and largest bars?

Answer:

Visual effect: (8-2)/2 = 300% increase in bar height
Data effect: (58-52)/52 ‚âà 11.5% increase
Lie Factor ‚âà 26 ‚Äî grossly misleading!

Grammar of Graphics: Altair Example

# 1. DATA

    .mark_bar()                    # 3. GEOMETRY

    )

Prefer OO over pyplot

PART III

D3.js

Data-Driven Documents

D3 provides the ultimate control for web-based visualisations. Higher learning curve, but unlimited flexibility.

The D3 Philosophy

D3 is not a charting library. It is a library for DOM manipulation based on data.

‚Äî You can create any visualisation imaginable, but you must build every element yourself. Maximum power, maximum effort.

Core Concepts

  • Selections: jQuery-like DOM manipulation
  • Data binding: Associate data with elements
  • Enter-Update-Exit: Handle data changes
  • Scales: Map data domains to visual ranges
  • Transitions: Animate between states
D3 operates at a lower level than charting libraries. This gives power but requires understanding the fundamentals.

The Enter-Update-Exit Pattern


// Select elements that (may) not exist yet
const circles = svg.selectAll('circle')
    .data(data, d => d.id);  // Key function for identity

// ENTER: New elements for new data
const enter = circles.enter()
    .append('circle')
    .attr('r', 0)  // Start small for animation
    .attr('fill', '#58a6ff');

// UPDATE: Existing elements
circles.merge(enter)
    .transition()
    .duration(750)
    .attr('cx', d => xScale(d.x))
    .attr('cy', d => yScale(d.y))
    .attr('r', 5);

// EXIT: Remove elements for deleted data
circles.exit()
    .transition()
    .duration(500)
    .attr('r', 0)
    .remove();
This pattern is fundamental to D3. It handles the three cases: new data, updated data and removed data.

D3 Scales: Mapping Domains


// Linear scale: continuous data ‚Üí pixels
const xScale = d3.scaleLinear()
    .domain([0, d3.max(data, d => d.value)])  // Input: data domain
    .range([margin.left, width - margin.right]); // Output: pixels

// Time scale: dates ‚Üí pixels
const timeScale = d3.scaleTime()
    .domain([new Date('2020-01-01'), new Date('2024-12-31')])
    .range([0, width]);

// Ordinal scale: categories ‚Üí colours
const colorScale = d3.scaleOrdinal()
    .domain(['A', 'B', 'C'])
    .range(d3.schemeCategory10);

// Band scale: categories ‚Üí positions (for bar charts)
const bandScale = d3.scaleBand()
    .domain(categories)
    .range([0, width])
    .padding(0.1);

// Usage
const cx = xScale(dataPoint.value);
const fill = colorScale(dataPoint.category);
Scales are the bridge between data space and visual space. Choose the appropriate scale for your data type.

Animated Transitions


function update(newData) {
    const bars = svg.selectAll('rect')
        .data(newData, d => d.id);
    
    // Enter: new bars start from bottom
    bars.enter()
        .append('rect')
        .attr('x', d => xScale(d.category))
        .attr('width', xScale.bandwidth())
        .attr('y', height)
        .attr('height', 0)
        .attr('fill', d => colorScale(d.category))
      .merge(bars)
      .transition()
        .duration(750)
        .ease(d3.easeCubicOut)
        .attr('y', d => yScale(d.value))
        .attr('height', d => height - yScale(d.value));
    
    // Exit: bars shrink and disappear
    bars.exit()
      .transition()
        .duration(500)
        .attr('height', 0)
        .attr('y', height)
        .remove();
}
Transitions make data updates visually comprehensible. The easing function controls the animation curve.

D3: Complete Line Chart


const margin = {top: 20, right: 20, bottom: 30, left: 50};
const width = 600 - margin.left - margin.right;
const height = 400 - margin.top - margin.bottom;

const svg = d3.select('#chart')

    .append('g')
    .attr('transform', `translate(${margin.left},${margin.top})`);

// Load data
d3.csv('data.csv').then(data => {
    const x = d3.scaleTime().domain(d3.extent(data, d => d.date)).range([0, width]);
    const y = d3.scaleLinear().domain([0, d3.max(data, d => d.value)]).range([height, 0]);
    
    // Axes
    svg.append('g').attr('transform', `translate(0,${height})`).call(d3.axisBottom(x));
    svg.append('g').call(d3.axisLeft(y));
    
    // Line
    const line = d3.line().x(d => x(d.date)).y(d => y(d.value));
    svg.append('path').datum(data).attr('d', line)
       .attr('fill', 'none').attr('stroke', '#58a6ff').attr('stroke-width', 2);
});
This is a minimal but complete D3 example. Note the viewBox for responsive sizing.

PART IV

Interactive Dashboards

Streamlit ‚Ä¢ Plotly Dash ‚Ä¢ Panel

Dashboards enable exploratory analysis that static figures cannot support.

Dashboard Design Principles

1. Overview First

Show summary metrics prominently. KPIs at the top.

2. Filter, Then Detail

Allow users to narrow scope before drilling down.

3. Linked Views

Selection in one chart updates all others.

"Overview first, zoom and filter, then details on demand"
‚Äî Shneiderman's Visual Information Seeking Mantra

Shneiderman's mantra is the foundation of dashboard design. Every dashboard should follow this pattern.

Streamlit: Rapid Dashboard Development


import streamlit as st
import pandas as pd
import plotly.express as px

st.title("Simulation Results Explorer")

# Sidebar filters
st.sidebar.header("Filters")
method = st.sidebar.selectbox("Method", ['Monte Carlo', 'RK4', 'Euler'])
n_range = st.sidebar.slider("Sample Size", 100, 10000, (1000, 5000))

# Load and filter data
@st.cache_data
def load_data():
    return pd.read_csv("simulation_results.csv")

data = load_data()
filtered = data[(data['method'] == method) & 
                (data['n'] >= n_range[0]) & (data['n'] <= n_range[1])]

# Layout
col1, col2 = st.columns(2)
with col1:
    st.metric("Mean Convergence", f"{filtered['value'].mean():.4f}")
    fig1 = px.histogram(filtered, x='value', nbins=50)
    st.plotly_chart(fig1, use_container_width=True)

with col2:
    st.metric("Std Deviation", f"{filtered['value'].std():.4f}")
    fig2 = px.scatter(filtered, x='n', y='value', color='method')
    st.plotly_chart(fig2, use_container_width=True)

Multi-Format Export


def save_publication_figure(
    fig: plt.Figure,
    name: str,
    output_dir: Path,
    formats: list[str] = ['pdf', 'png', 'svg']
) -> dict[str, Path]:
    """Save figure in multiple formats for publication.
    
    Args:
        fig: Matplotlib figure object
        name: Base filename (without extension)
        output_dir: Directory for output files
        formats: List of formats to export
        
    Returns:
        Dictionary mapping format to saved path
    """
    output_dir.mkdir(parents=True, exist_ok=True)
    paths = {}
    
    for fmt in formats:
        path = output_dir / f'{name}.{fmt}'
        fig.savefig(
            path,
            format=fmt,
            dpi=300 if fmt == 'png' else None,  # DPI only for raster
            bbox_inches='tight',
            pad_inches=0.05,
            facecolor='white',
            edgecolor='none'
        )
        paths[fmt] = path
    
    return paths
Export to multiple formats: PDF for journals, PNG for presentations, SVG for web/editing. This function handles all cases.
